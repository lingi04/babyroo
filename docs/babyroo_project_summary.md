# Babyroo 프로젝트 정리

## 1. 프로젝트 한 줄 정의
**아이와 뭐 할지 모르는 육아초보 아빠를 위한 0~3세 외출 추천 앱**

육아초보 아빠가 여러 사이트를 돌아다니거나 검색어를 고민하지 않고도,
**아이 월령과 오늘의 상황에 맞는 외출 후보를 빠르게 고르고 실행할 수 있게 하는 것**이 핵심입니다.

1차 성공 장면은 금요일 밤이나 토요일 아침에 서울/경기 거주 초보 아빠가 0~36개월 아이와 당일 또는 이번 주말에 갈 만한 후보 3개를 3분 안에 고르고, 원문 예약 페이지로 이동하는 것입니다.

## 2. 해결하려는 문제

지금 0~3세 아기가 갈 수 있는 정보는 여기저기 흩어져 있습니다.

- 공공 문화행사 사이트
- 구청 / 육아종합지원센터
- 어린이회관 / 박물관 / 체험관
- 민간 예매처
- 블로그 / 맘카페 / SNS

문제는 정보가 없는 것이 아니라,
**부모 입장에서 찾기 불편하게 흩어져 있고 정리가 안 되어 있다는 점**입니다.

특히 육아초보 아빠는 보통 이런 정보가 부족해서 결정을 미룹니다.

- 우리 아이 개월수에 맞는가
- 보호자 동반이 필요한가
- 공연형인지, 뛰어놀 수 있는 체험형인지
- 실내인지
- 유모차 / 수유실 / 주차가 되는지
- 예약 링크가 어디인지
- 이번 주말에 갈 만한 곳이 있는지
- 처음 가도 실패할 가능성이 낮은지
- 준비물이 복잡하지 않은지

즉, Babyroo는
**행사 정보 자체를 생산하는 서비스가 아니라, 육아초보 아빠가 아이와 뭘 할지 결정하게 도와주는 추천 서비스**입니다.

## 3. 서비스 방향

### 핵심 컨셉
- **0~3세 중심**
- **육아초보 아빠 관점의 외출 결정 지원**
- **공연 / 체험 / 실내놀이 정보 통합**
- **빠르게 고르고 바로 원문/예약으로 이동**

### 차별점
기존 사이트들은 보통 “아동”, “가족”, “문화행사” 단위로 보여주지만,
Babyroo는 더 실용적으로 정리합니다.

예:
- 12개월 이상
- 24개월 이상
- 보호자 동반 필수
- 실내
- 비 오는 날 좋음
- 짧은 체험
- 처음 가기 좋음
- 주차 편함
- 예약만 하면 됨
- 1시간 코스
- 무료
- 사전예약 필요

즉, **월령 중심 + 아빠의 결정 부담을 줄이는 실전 태그**가 핵심 차별점입니다.

## 4. 초기 MVP 범위

처음부터 크게 가지 않고, 아주 좁게 시작합니다.

### 지역
- 서울
- 경기 일부

### 대상 연령
- 0~3세

### 카테고리
- 공연
- 체험
- 실내놀이터 / 놀이공간

### 초기 목표
- 정보 수집
- 정리
- 추천하기
- 보여주기

즉, 아직은 커뮤니티나 예약 플랫폼이 아니라
**아이와 뭐 할지 빠르게 정하게 해주는 외출 추천 앱**으로 시작합니다.

초기 홈 경험은 전체 행사 탐색보다 이번 주말 후보 고르기에 가깝게 둡니다. 전체 목록과 필터는 보조 기능이고, 핵심은 아이 월령, 지역, 날짜, 상황 조건을 바탕으로 후보를 압축하는 것입니다.

## 5. 기술 방향

현재 방향은 아래와 같습니다.

- **백엔드 없음**
- **크롤러**
- **정적 웹**
- **S3 호스팅**

이 방향은 MVP에 잘 맞습니다.

초기에는 API 서버 없이도 충분하며,
**JSON 파일이 곧 데이터 API 역할**을 하게 됩니다.

## 6. 예상 아키텍처

### 구성 요소
1. **크롤러**
   - 여러 사이트에서 행사 정보 수집
   - 원문 저장
   - 정규화 처리
   - 최종 JSON 생성

2. **정적 웹**
   - 행사 목록 표시
   - 필터/검색 제공
   - 상황별 추천 제공
   - 상세 페이지 제공

3. **호스팅**
   - S3
   - 필요 시 CloudFront

### 데이터 흐름
1. 크롤러 실행
2. 각 사이트 수집
3. 표준 스키마로 정리
4. `events.json` 생성
5. 정적 웹에서 JSON fetch
6. 사용자에게 목록/상세 노출

즉, **서버 없는 읽기 중심 구조**입니다.

## 7. 데이터 표준화 방향

각 사이트마다 정보 형식이 달라서,
Babyroo의 핵심은 크롤링 자체보다 **정규화**입니다.

예를 들어 이런 값들이 제각각입니다.

- “24개월 이상”
- “영유아 및 보호자 참여”
- “미취학 아동”
- “36개월 미만은 보호자 밀착 케어 필요”

이걸 Babyroo 내부의 공통 필드로 바꿔야 합니다.

예:
- `age_min_months`
- `age_max_months`
- `guardian_required`
- `category`
- `price_type`
- `tags`

## 8. LLM 활용 아이디어

**LLM으로 표준화 보조를 하는 것은 가능하고, 잘 맞는 작업입니다.**
하지만 전부 맡기기보다는 **규칙 기반 + LLM 보조**가 가장 안정적입니다.

### 코드로 처리할 것
- 날짜 파싱
- 가격 숫자 추출
- URL 추출
- 주소 정리
- 종료 행사 제거
- 중복 후보 계산

### LLM으로 처리할 것
- 연령 정보 해석
- 카테고리 분류
- 아빠 관점의 실전 태그 생성
- 요약 생성
- 비정형 텍스트에서 편의 정보 추출

### 추천 원칙
- JSON Schema 기반 구조화 출력
- 모르면 `null`
- 추측 금지
- `confidence` 반환
- 가능하면 `evidence`도 저장

즉, LLM은 “자동 분류기 + 정리 도우미” 역할입니다.

## 9. 데이터 파이프라인 권장 구조

다음 구조를 권장합니다.

- `raw/`
  - 크롤러가 가져온 원문

- `normalized/`
  - 정규화된 중간 결과

- `public/`
  - 웹에서 사용하는 최종 JSON

이렇게 나누면 나중에
- 파서가 잘못됐는지
- LLM이 잘못 해석했는지
- 최종 공개 단계에서 필터링됐는지
추적하기 쉬워집니다.

### 데이터 품질 우선순위

수집원을 늘리기 전에 아래 기준을 먼저 고정합니다.

- 공개 가능한 행사의 최소 필수값
- 필터용 값의 `true` / `false` / `null` 판단 기준
- 종료 행사 제외 기준
- 중복 행사 후보 판단 기준
- LLM 보조 결과의 `confidence`와 `evidence` 저장 위치

초기에는 많은 행사를 모으는 것보다, 적은 행사라도 육아초보 아빠가 믿고 바로 고를 수 있게 만드는 것을 우선합니다.

## 12. 현재 프로젝트 포지션

이 프로젝트는 아직 **검증 전 단계의 MVP 아이디어**이며,
성공 여부는 결국 아래에 달려 있습니다.

- 부모가 실제로 자주 쓰는가
- 월령 기반 필터가 진짜 유용한가
- 정보 신뢰도가 충분한가
- “이번 주말 뭐하지” 문제를 얼마나 잘 해결하는가

즉, 지금은 복잡한 기술보다
**좋은 정보 구조 + 실제 쓸 만한 데이터 품질**이 더 중요합니다.

## 13. 브랜드명

프로젝트명은 **Babyroo**로 결정했습니다.

### Babyroo 느낌
- 부드럽고 귀여움
- 기억하기 쉬움
- 브랜드화하기 좋음
- 아기/육아 서비스와 잘 어울림
- 캥거루/주머니/외출 같은 이미지 확장 가능

즉, 기능 설명형 이름보다는
**브랜드형 이름**으로 가는 방향입니다.

## 14. 핵심 결론

### 프로젝트 정의
Babyroo는
**0~3세 아기와 보호자를 위한 공연·체험 정보 큐레이션 웹 서비스**입니다.

### MVP 전략
처음에는
- 서울/경기 일부
- 공연/체험/실내놀이
- 크롤러 + 정적 웹 + S3
- 백엔드 없음

으로 작게 시작합니다.

### 기술 전략
- 데이터 수집은 크롤러
- 웹은 정적 사이트
- 공개 데이터는 JSON
- 애매한 정보 정규화는 LLM 보조

### 제품 전략
핵심 경쟁력은
- **0~3세 월령 기준 탐색**
- **부모 관점 태그**
- **흩어진 정보의 통합 정리**
입니다.
