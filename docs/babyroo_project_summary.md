# Babyroo 프로젝트 정리

## 1. 프로젝트 한 줄 정의
**아이와 뭐 할지 모르는 육아초보 아빠를 위한 0~3세 외출 추천 앱**

육아초보 아빠가 여러 사이트를 돌아다니거나 검색어를 고민하지 않고도,
**아이 월령과 오늘의 상황에 맞는 외출 후보를 빠르게 고르고 실행할 수 있게 하는 것**이 핵심입니다.

1차 성공 장면은 금요일 밤이나 토요일 아침에 서울/경기 거주 초보 아빠가 0~36개월 아이와 당일 또는 이번 주말에 갈 만한 후보 3개를 3분 안에 고르고, 원문 예약 페이지로 이동하는 것입니다.

## 2. 해결하려는 문제

지금 0~3세 아기가 갈 수 있는 정보는 여기저기 흩어져 있습니다.

- 공공 문화행사 사이트
- 구청 / 육아종합지원센터
- 어린이회관 / 박물관 / 체험관
- 민간 예매처
- 블로그 / 맘카페 / SNS

문제는 정보가 없는 것이 아니라,
**부모 입장에서 찾기 불편하게 흩어져 있고 정리가 안 되어 있다는 점**입니다.

특히 육아초보 아빠는 보통 이런 정보가 부족해서 결정을 미룹니다.

- 우리 아이 개월수에 맞는가
- 보호자 동반이 필요한가
- 공연형인지, 뛰어놀 수 있는 체험형인지
- 실내인지
- 유모차 / 수유실 / 주차가 되는지
- 예약 링크가 어디인지
- 이번 주말에 갈 만한 곳이 있는지
- 처음 가도 실패할 가능성이 낮은지
- 준비물이 복잡하지 않은지

즉, Babyroo는
**행사 정보 자체를 생산하는 서비스가 아니라, 육아초보 아빠가 아이와 뭘 할지 결정하게 도와주는 추천 서비스**입니다.

## 3. 서비스 방향

### 핵심 컨셉
- **0~3세 중심**
- **육아초보 아빠 관점의 외출 결정 지원**
- **공연 / 체험 / 실내놀이 정보 통합**
- **빠르게 고르고 바로 원문/예약으로 이동**

### 차별점
기존 사이트들은 보통 “아동”, “가족”, “문화행사” 단위로 보여주지만,
Babyroo는 더 실용적으로 정리합니다.

예:
- 12개월 이상
- 24개월 이상
- 보호자 동반 필수
- 실내
- 비 오는 날 좋음
- 짧은 체험
- 처음 가기 좋음
- 주차 편함
- 예약만 하면 됨
- 1시간 코스
- 무료
- 사전예약 필요

즉, **월령 중심 + 아빠의 결정 부담을 줄이는 실전 태그**가 핵심 차별점입니다.

## 4. 초기 MVP 범위

처음부터 크게 가지 않고, 아주 좁게 시작합니다.

### 지역
- 서울
- 경기 일부

### 대상 연령
- 0~3세

### 카테고리
- 공연
- 체험
- 실내놀이터 / 놀이공간

### 초기 목표
- 정보 수집
- 정리
- 추천하기
- 보여주기

즉, 아직은 커뮤니티나 예약 플랫폼이 아니라
**아이와 뭐 할지 빠르게 정하게 해주는 외출 추천 앱**으로 시작합니다.

초기 홈 경험은 전체 행사 탐색보다 이번 주말 후보 고르기에 가깝게 둡니다. 전체 목록과 필터는 보조 기능이고, 핵심은 아이 월령, 지역, 날짜, 상황 조건을 바탕으로 후보를 압축하는 것입니다.

## 5. 기술 방향

현재 MVP 개발 순서는 아래처럼 조정합니다.

- **클라이언트 앱 우선**
- **더미 데이터 기반 개발**
- **백엔드 없음**
- **React Native 앱 우선**
- **iOS / Android 우선, Web은 보조 타깃**
- **이후 크롤러 연결**

크롤러 개발 난도가 예상보다 높기 때문에, 초기에 크롤러를 완성한 뒤 화면을 만드는 순서는 리스크가 큽니다.
먼저 `admin/resources/events.csv`를 더미 데이터이자 화면 개발용 데이터 계약으로 두고,
사용자가 실제로 보게 될 추천/목록/상세 경험을 먼저 만듭니다.

초기에는 API 서버 없이도 충분하며, CSV 또는 CSV에서 변환한 JSON 파일이 데이터 API 역할을 합니다.
클라이언트는 **React Native 기반 모바일 앱**으로 먼저 만들고,
Expo 없이 React Native CLI 기반으로 개발합니다.
웹은 같은 제품 경험을 확인하는 보조 타깃으로 둘 수 있지만, Babyroo의 1차 기준은 모바일 앱입니다.
중요한 것은 크롤러 완성도가 아니라, **부모가 3분 안에 후보를 고를 수 있는 클라이언트 경험을 먼저 검증하는 것**입니다.

## 6. 예상 아키텍처

### 구성 요소
1. **React Native 모바일 앱**
   - 앱 코드는 `app/` 디렉터리에 둠
   - 행사 목록 표시
   - 필터/검색 제공
   - 상황별 추천 제공
   - 상세 페이지 제공
   - 원문/예약 링크 제공
   - 엄지 조작에 맞는 카드/버튼/필터 UI 제공
   - 이동 중, 외출 직전, 주말 아침에 빠르게 고르는 사용 맥락에 최적화
   - iOS와 Android를 같은 코드베이스에서 제공
   - 필요하면 별도 웹 클라이언트로 web 타깃 확장

2. **더미 데이터**
   - `admin/resources/events.csv`를 초기 데이터 소스로 사용
   - 화면에 필요한 필드와 추천 정책을 먼저 고정
   - CSV의 아래쪽 레코드를 더 나중에 추가된 데이터로 간주
   - `admin/`은 운영/리뷰 도구로 유지하고, 사용자-facing 앱은 `app/`에서 개발

3. **크롤러**
   - 여러 사이트에서 행사 정보 수집
   - 원문 저장
   - 정규화 처리
   - 더미 CSV/공개 JSON과 같은 스키마로 데이터 생성

4. **배포**
   - iOS: TestFlight 이후 App Store
   - Android: 내부 테스트 이후 Play Store
   - Web 타깃을 유지한다면 S3/CloudFront 같은 정적 호스팅 사용 가능

### 1차 데이터 흐름: 클라이언트 우선
1. 사람이 `admin/resources/events.csv`에 더미 또는 수동 검증 데이터를 추가
2. 클라이언트 앱이 CSV 또는 변환된 JSON을 읽음
3. 추천 정책, 필터, 목록, 상세 화면을 구현
4. 사용자 관점에서 정보 구조와 결정 경험을 검증

### 2차 데이터 흐름: 크롤러 연결
1. 크롤러 실행
2. 각 사이트 수집
3. 표준 스키마로 정리
4. `events.json` 또는 `events.csv` 생성
5. React Native 앱에서 번들된 데이터 또는 원격 데이터 파일을 읽음
6. 사용자에게 목록/상세 노출

즉, **서버 없는 읽기 중심 구조**입니다.
다만 MVP의 첫 병목은 크롤러가 아니라 클라이언트 경험 검증입니다.

### React Native 선택

Babyroo의 주 사용 환경은 모바일입니다.
따라서 MVP 구현도 모바일 앱 경험을 1차 기준으로 둡니다.

- **제품 경험**: iOS/Android 모바일 앱처럼 설계합니다.
- **기술 구현**: Expo 없이 React Native CLI를 사용합니다.
- **개발 환경**: Metro, Android Studio, Xcode를 기본 개발 도구로 둡니다.
- **이유**: 한 코드베이스로 iOS와 Android를 만들면서도 네이티브 프로젝트 설정을 직접 제어할 수 있습니다.
- **Web 타깃**: 필요하면 별도 웹 클라이언트로 대응합니다. 단, MVP 판단 기준은 모바일 앱 사용성입니다.

즉, 이전의 “정적 웹앱 우선” 방향은 “React Native 모바일 앱 우선”으로 바뀝니다.
정적 호스팅은 Web 타깃이 필요할 때의 배포 방식으로만 남깁니다.

## 7. 데이터 표준화 방향

각 사이트마다 정보 형식이 달라서,
Babyroo의 핵심은 크롤링 자체보다 **정규화**입니다.

예를 들어 이런 값들이 제각각입니다.

- “24개월 이상”
- “영유아 및 보호자 참여”
- “미취학 아동”
- “36개월 미만은 보호자 밀착 케어 필요”

이걸 Babyroo 내부의 공통 필드로 바꿔야 합니다.

예:
- `age_min_months`
- `age_max_months`
- `guardian_required`
- `category`
- `price_type`
- `tags`

## 8. LLM 활용 아이디어

**LLM으로 표준화 보조를 하는 것은 가능하고, 잘 맞는 작업입니다.**
하지만 전부 맡기기보다는 **규칙 기반 + LLM 보조**가 가장 안정적입니다.

### 코드로 처리할 것
- 날짜 파싱
- 가격 숫자 추출
- URL 추출
- 주소 정리
- 종료 행사 제거
- 중복 후보 계산

### LLM으로 처리할 것
- 연령 정보 해석
- 카테고리 분류
- 아빠 관점의 실전 태그 생성
- 요약 생성
- 비정형 텍스트에서 편의 정보 추출

### 추천 원칙
- JSON Schema 기반 구조화 출력
- 모르면 `null`
- 추측 금지
- `confidence` 반환
- 가능하면 `evidence`도 저장

즉, LLM은 “자동 분류기 + 정리 도우미” 역할입니다.

## 9. 데이터 파이프라인 권장 구조

다음 구조를 권장합니다.

- `raw/`
  - 크롤러가 가져온 원문

- `normalized/`
  - 정규화된 중간 결과

- `public/`
  - 웹에서 사용하는 최종 JSON

이렇게 나누면 나중에
- 파서가 잘못됐는지
- LLM이 잘못 해석했는지
- 최종 공개 단계에서 필터링됐는지
추적하기 쉬워집니다.

### 데이터 품질 우선순위

수집원을 늘리기 전에 아래 기준을 먼저 고정합니다.

- 공개 가능한 행사의 최소 필수값
- 필터용 값의 `true` / `false` / `null` 판단 기준
- 종료 행사 제외 기준
- 중복 행사 후보 판단 기준
- LLM 보조 결과의 `confidence`와 `evidence` 저장 위치

초기에는 많은 행사를 모으는 것보다, 적은 행사라도 육아초보 아빠가 믿고 바로 고를 수 있게 만드는 것을 우선합니다.

## 11. 화면 구조 초안

화면은 데스크톱보다 모바일을 우선합니다.
카드는 한 손으로 스크롤하며 볼 수 있어야 하고, 주요 필터는 작은 화면에서도 빠르게 바꿀 수 있어야 합니다.

### 홈
- 오늘/이번 주말 추천
- 빠른 상황 선택
- 최신 행사 목록
- 아이 월령과 지역을 가장 먼저 입력/확인

### 목록 페이지
- 모바일 필터 바 또는 바텀시트
- 추천 이유 표시
- 행사 카드 리스트
- 정렬

### 상세 페이지
- 행사 요약
- 추천 이유
- 날짜/장소
- 권장 연령
- 보호자 동반 여부
- 편의시설 정보
- 원문 링크
- 마지막 확인일

### 모바일 UX 원칙
- 첫 화면에서 바로 후보를 보여주고, 긴 설명보다 선택 가능한 카드에 집중합니다.
- 필터는 상단 고정 또는 바텀시트로 제공해 목록으로 쉽게 돌아오게 합니다.
- 카드의 주요 액션은 엄지로 누르기 쉬운 위치에 둡니다.
- 원문/예약 링크는 상세 하단뿐 아니라 카드 또는 상세 상단에서도 접근 가능하게 둡니다.
- 데스크톱 레이아웃은 모바일 경험을 확장한 보조 화면으로 봅니다.

## 12. 현재 프로젝트 포지션

이 프로젝트는 아직 **검증 전 단계의 MVP 아이디어**이며,
성공 여부는 결국 아래에 달려 있습니다.

- 부모가 실제로 자주 쓰는가
- 월령 기반 필터가 진짜 유용한가
- 정보 신뢰도가 충분한가
- “이번 주말 뭐하지” 문제를 얼마나 잘 해결하는가

즉, 지금은 복잡한 기술보다
**좋은 정보 구조 + 실제 쓸 만한 데이터 품질**이 더 중요합니다.

따라서 개발 순서는 크롤러 중심에서 클라이언트 중심으로 바꿉니다.
수집 자동화는 중요하지만, 먼저 확인해야 할 것은 아래 질문입니다.

- 더미 데이터만 있어도 부모가 후보를 빠르게 고를 수 있는가
- 월령/지역/날짜/상황 필터가 화면에서 자연스럽게 작동하는가
- 행사 카드와 상세 화면이 충분한 결정을 돕는가
- 추천 이유가 신뢰 가능하고 과장 없이 보이는가
- 어떤 필드가 화면에서 정말 필요하고, 어떤 필드는 후순위인가

- 부드럽고 귀여움
- 기억하기 쉬움
- 브랜드화하기 좋음
- 아기/육아 서비스와 잘 어울림
- 캥거루/주머니/외출 같은 이미지 확장 가능

즉, 기능 설명형 이름보다는
**브랜드형 이름**으로 가는 방향입니다.

## 14. 핵심 결론

### 프로젝트 정의
Babyroo는
**0~3세 아기와 보호자를 위한 공연·체험 정보 큐레이션 웹 서비스**입니다.

### MVP 전략
처음에는
- 서울/경기 일부
- 공연/체험/실내놀이
- 더미 데이터 기반 React Native 모바일 앱
- 백엔드 없음

으로 작게 시작합니다.

### 기술 전략
- 먼저 `admin/resources/events.csv`를 기준으로 화면과 추천 흐름을 구현
- React Native 앱 코드는 `app/` 디렉터리에 둠
- 앱은 Expo 없이 React Native CLI로 구현
- Android Studio와 Xcode 기반으로 iOS/Android 빌드와 실행 환경을 구성
- Web 타깃은 보조로 고려하되 MVP의 1차 기준은 모바일 앱 사용성
- 공개 데이터는 CSV 또는 CSV에서 변환한 JSON
- 크롤러는 후속 단계에서 같은 스키마를 채우는 데이터 생산자로 연결
- 애매한 정보 정규화는 후속 단계에서 규칙 기반 + LLM 보조로 처리

### 제품 전략
핵심 경쟁력은
- **0~3세 월령 기준 탐색**
- **부모 관점 태그**
- **흩어진 정보의 통합 정리**
입니다.

## 15. 다음 단계

이제 다음 단계는 아래 순서가 자연스럽습니다.

1. **클라이언트 앱 첫 화면 구현**
   - 홈에서 이번 주말/오늘 후보를 바로 보여줌
   - 전체 탐색보다 빠른 후보 압축 경험을 우선함
   - 모바일 화면을 기준으로 먼저 설계함

2. **`admin/resources/events.csv` 기반 데이터 로딩**
   - CSV를 초기 데이터 계약으로 사용
   - 아래쪽 행이 더 나중에 추가된 데이터라고 보고 최신순으로 표시
   - 카드 순번은 원래 CSV 행 순서를 보존해 `5, 4, 3, 2, 1`처럼 보이게 함

3. **추천 정책 구현**
   - 월령, 날짜, 지역, 카테고리, 태그 기반 추천
   - 추천 이유는 규칙 기반 문장으로 생성

4. **목록/상세/필터 경험 완성**
   - 행사 카드
   - 상세 화면
   - 원문/예약 링크
   - 지역/월령/날짜/무료/카테고리 필터
   - 모바일 필터 바 또는 바텀시트

5. **데이터 스키마 보강**
   - 실제 화면을 만들면서 필요한 필드와 불필요한 필드를 구분
   - CSV/JSON 계약을 안정화

6. **크롤러 재개**
   - 수집 대상 사이트 1차 선정
   - 첫 실제 collector 구현
   - 크롤러 산출물이 클라이언트 데이터 계약을 채우게 함
